## 3. Process Management 1 & 2: 프로세스 관리 및 자원 분류

### 1) 프로세스 생성 (Process Creation)
* **원리:** 부모 프로세스(Parent Process)가 자식 프로세스(Child Process)를 복제 생성하는 트리(Tree) 구조 형태를 가집니다.
* **주소 공간 (Address Space):**
  * 자식은 부모의 주소 공간을 그대로 복사(Duplicate)합니다.
  * 자식은 복사된 공간 위에 새로운 프로그램의 이미지를 덮어씌워(Load) 실행합니다.
* **Unix/Linux에서의 대표적 시스템 콜:**
  * `fork()`: 부모 프로세스를 그대로 복사하여 새로운 자식 프로세스를 생성합니다. (PID를 제외한 모든 문맥 복사)
  * `exec()`: 새로운 프로그램을 메모리에 덮어씌워 완전히 새로운 작업을 시작하게 만듭니다. -> fork를 하는게 아니라 완전히 새로운 요리책을 머리속에 집어넣는 시스템 콜이라고 이해해도 좋다.
  * `wait()`: 자식 프로세스가 종료될 때까지 부모 프로세스를 대기(Blocked) 상태로 둡니다.
  * `exit()`: 프로세스를 자발적으로 종료하고 자원을 반납하며 부모에게 종료 상태를 전달합니다. -> 자식이 자발적으로 부모에게 끝났다고 전달하는 시스템 콜임.

### 2) 프로세스 종료 (Process Termination)
* **자발적 종료:** 프로세스가 마지막 명령을 수행한 후 `exit()` 시스템 콜을 호출하여 OS에 종료를 알립니다.
* **비자발적 종료 (Abort / 강제 종료):** 부모 프로세스가 자식 프로세스를 `abort()`하는 경우입니다.
  * 자식이 할당된 자원의 한계를 초과하여 사용할 때.
  * 자식에게 할당된 태스크가 더 이상 필요하지 않을 때.
  * 부모 프로세스가 종료(Exit)될 때 (일부 OS는 부모가 죽으면 자식을 모두 단계적으로 강제 종료시키는 **Cascading Termination**을 수행).
    - > 메인 셰프(부모)가 보조 요리사에게 "야! 손님 나갔어, 양파 그만 썰고 당장 퇴근해!"라고 강제로 명령을 내리는 것이 kill()입니다. (정확히는 강제 종료 '신호'를 보내는 것입니다.)

    - > 그렇다면 abort()는 정확히 언제 쓰나요?
      abort()도 비정상적으로 종료된다는 점에서는 비슷하지만, 부모가 자식을 죽이는 것이 아니라 자식 프로세스 스스로 "치명적인 실수"를 저질러서 강제로 자폭(비정상 종료)하는 것을 의미합니다.

      - 상황: 보조 요리사(자식)가 양파를 썰다가, 실수로 자기 손가락을 베어버렸습니다. (컴퓨터로 치면 '0으로 나누기 오류'나 '잘못된 메모리 접근' 같은 심각한 버그 발생)

      - 행동: 보조 요리사가 "아악! 피 난다! 나 요리 못해!" 하고 하던 일을 다 내팽개치고 응급실로 도망가 버리는(스스로 비정상 종료) 것이 바로 abort()입니다.

  -> 즉: exit() (정상 종료): 자식이 스스로 할 일을 다 끝내고 "저 다 했어요, 퇴근할게요!" 하는 것.

    - kill() (타의에 의한 강제 종료): 부모(또는 OS)가 자식에게 "너 당장 하던 거 멈추고 나가!" 하고 해고하는 것.

    - abort() (자폭, 비정상 종료): 자식 스스로 치명적인 오류를 일으켜 "앗, 망했다! 더 이상 못해!" 하고 강제로 터져버리는 것.

### 3) 프로세스 간 협력 (Interprocess Communication, IPC)
* **독립적 프로세스 (Independent Process):** 프로세스들이 메모리 공간을 공유하지 않으므로 서로 영향을 주지 않고 독자적으로 수행됩니다.
* **협력 프로세스 (Cooperating Process):** 타 프로세스의 수행에 영향을 주거나 받는 프로세스입니다. 효율성 증대를 위해 IPC 메커니즘을 사용합니다.
* **IPC 모델 분류:**
  1. **메시지 전달 (Message Passing):**
     * 프로세스 간 공유 메모리 없이 **커널(Kernel)**을 통해 메시지를 주고받는 안전한 방식입니다.
     * *Direct Communication:* 통신하려는 프로세스의 이름을 명시적으로 지정하여 전달 (`send(P, message)`).
     * *Indirect Communication:* 별도의 메일박스(Mailbox) 또는 포트(Port)를 통해 메시지를 간접 전달.
  2. **공유 메모리 (Shared Memory):**
     * 커널에 시스템 콜을 보내 두 프로세스가 주소 공간의 일부를 공유하도록 설정합니다.
     * 한 번 설정되면 커널의 중재 없이 메모리에 직접 접근하므로 **통신 속도가 매우 빠르지만**, 데이터 일관성을 위해 동기화(Synchronization) 문제를 자체적으로 해결해야 합니다.

### 4) CPU 및 I/O 버스트 (CPU and I/O Bursts)
프로그램이 실행될 때는 CPU 계산을 수행하는 시간(**CPU Burst**)과 파일 입력, 네트워크 송수신, 화면 출력 등을 기다리는 시간(**I/O Burst**)이 번갈아 나타납니다.

* **CPU-burst Time의 분포 특성:**
  * 대다수의 프로세스는 **짧은 CPU Burst**를 자주 가지고, 소수의 프로세스만 **매우 긴 CPU Burst**를 가집니다.
* **프로세스의 특성 분류:**
  * **I/O-bound Process:** CPU를 짧게 쓰고 대다수 시간을 I/O 작업에 소모하는 프로세스 (예: 인터랙티브 웹 앱, 워드프로세서). 짧은 CPU Burst를 가집니다.
  * **CPU-bound Process:** 대규모 연산이나 계산 위주의 작업을 수행하는 프로세스 (예: 기상 예측 시뮬레이션, AI 모델 학습). 길고 연속적인 CPU Burst를 가집니다.

### 5) CPU 스케줄러 & 디스패처 (CPU Scheduler & Dispatcher)
* **CPU Scheduler:** Ready 상태의 프로세스들 중에서 이번에 CPU를 어떤 프로세스에게 넘겨줄지 선택하는 운영체제의 커널 코드(알고리즘)입니다.
* **Dispatcher:** CPU 스케줄러가 선택한 프로세스에게 **실제로 CPU 제어권을 넘겨주는 역할을 하는 모듈**입니다. 문맥 교환(Context Switch)을 수행하고 유저 모드로 전환하여 프로그램의 해당 PC(Program Counter) 주소로 점프합니다.
  * *Dispatch Latency:* 디스패처가 하나의 프로세스를 멈추고 다른 프로세스를 실행하기까지 걸리는 시간으로, 시스템 성능을 위해 최소화되어야 합니다.
