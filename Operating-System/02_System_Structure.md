# 02. System Structure & Program Execution (시스템 구조와 프로그램 실행)

> **Context:** CPU는 압도적으로 빠르지만 스스로 외부 장치에 접근할 수 없다. 반면 I/O 장치(디스크, 키보드)는 매우 느리다. 만약 CPU가 I/O 작업이 끝날 때까지 기다린다면 시스템은 심각한 자원 낭비를 겪게 된다. 이를 해결하기 위해 등장한 개념이 **인터럽트(Interrupt) 기반 구조**와 **비동기적 처리**다.

## 2.1 컴퓨터 시스템의 기본 구조와 보호 메커니즘

CPU는 오직 메모리에 올라온 기계어 명령어(Instruction)만 순차적으로 읽어서 실행할 뿐이다. 

### 왜 제어권 방어 장치가 필요한가?
만약 사용자 프로그램이 무한 루프를 돌며 CPU를 독점하거나, 마음대로 디스크의 중요 파일을 삭제하려고 한다면? 이를 막기 위해 하드웨어적인 방어 장치들이 존재한다.

1. **Mode bit (모드 비트)**
   - **Why:** 운영체제와 사용자 프로그램을 구분하여 시스템을 보호하기 위함.
   - `0 (Kernel Mode)`: 모든 권한을 가짐. 시스템에 중대한 영향을 미치는 보안상 중요한 명령(특권 명령) 수행 가능.
   - `1 (User Mode)`: 제한된 권한. 일반적인 연산만 가능하며, I/O 접근 등의 특권 명령을 실행하려 하면 하드웨어가 차단하고 OS로 제어권을 넘김.
2. **Timer (타이머)**
   - **Why:** 특정 프로그램의 CPU 독점을 막기 위함 (시분할 시스템의 핵심 하드웨어).
   - 정해진 시간이 지나면 타이머가 CPU에 **인터럽트**를 걸어 제어권을 강제로 운영체제(OS)에게 넘기도록 한다.

### 🎤 [Backend Interview Point] 시스템 보호 메커니즘
**Q. 사용자 프로그램이 디스크의 파일을 읽어오려고 합니다. User Mode에서는 I/O 접근이 불가능한데, 이 과정이 어떻게 이루어지나요?**
- 사용자 프로그램은 직접 특권 명령을 실행할 수 없으므로, OS에게 부탁하기 위해 **System Call(시스템 콜)**을 호출합니다. 
- 시스템 콜이 호출되면 소프트웨어 인터럽트(Trap)가 발생하여 Mode bit가 1에서 0(Kernel Mode)으로 변경되고, CPU 제어권이 OS로 넘어갑니다.
- OS는 해당 I/O 요청이 유효한지 검사한 후, 디스크 컨트롤러에게 I/O를 지시합니다. I/O가 완료되면 다시 Mode bit를 1로 바꾸고 사용자 프로그램에게 결과를 반환합니다.

## 2.2 인터럽트(Interrupt)와 시스템 콜(System Call)

현대의 운영체제는 **인터럽트에 의해 구동(Interrupt-driven)**된다. OS는 평소에 대기(Sleep)하다가 하드웨어나 소프트웨어가 인터럽트를 걸어주면 그제서야 깨어나 CPU 제어권을 얻고 코드를 실행한다. (OS 스스로 인터럽트를 거는 것이 아님!)

- **하드웨어 인터럽트 (Asynchronous):** I/O 장치(디스크 컨트롤러, 키보드 등)나 Timer가 CPU에게 "잠깐 처리할 일이 생김"을 알리는 전기적 신호다. 명령어 실행 중 언제든 발생할 수 있다.
- **소프트웨어 인터럽트 / Trap (Synchronous):** 사용자 프로그램이 실행 중인 명령어 자체에 의해 발생한다.
  - **Exception (예외):** 프로그램이 0으로 나누거나(Divide by Zero), 권한 밖의 메모리를 참조(Segmentation Fault)하는 등 오류를 범했을 때 발생. (OS가 해당 프로세스를 강제 종료함)
  - **System Call (시스템 콜):** 권한이 없는 작업(I/O 등)을 위해 **자발적으로 OS에게 서비스를 요청**하기 위해 발생시키는 인터럽트.

**[인터럽트 처리 절차 Deep Dive]**
1. 인터럽트 발생 → 현재 CPU의 상태(PC, Registers)를 PCB에 저장.
2. 인터럽트 벡터(Interrupt Vector)를 참조하여, 발생한 인터럽트에 맞는 **인터럽트 처리 루틴(ISR: Interrupt Service Routine)** 주소를 찾음.
3. 해당 커널 코드를 실행.

## 2.3 [핵심] Timer Interrupt와 Context Switching 과정

### Timer Interrupt의 정의 및 필요성
- OS가 CPU 제어권을 강제로 다시 가져오기 위한 장치.
- 하드웨어 타이머가 일정 시간(수십 ms)마다 인터럽트를 발생시킨다.

**[왜 필요한가? - CPU 독점 방지]**
```c
while(true) {
    // 악의적이거나 버그로 인한 무한 루프
}

```


위와 같이 무한 루프가 돌아도 CPU를 영원히 점유하는 것은 불가능하다. Timer Interrupt가 걸리면 하드웨어적으로 CPU 제어권이 OS로 넘어가기 때문이다.

### Timer Interrupt의 상세 흐름
```text
1. 사용자 프로그램 실행 (Mode bit: 1)
↓
2. Timer interrupt 발생 (설정된 시간 만료)
↓
3. CPU 상태 저장 및 커널 모드 전환 (Mode bit: 0)
↓
4. OS의 Timer Interrupt Handler 실행
↓
5. OS 내부의 Scheduler(스케줄러) 실행
↓
6. CPU를 다른 프로세스에게 넘길지 결정 (Context Switching 발생)
```

### Context Switching (문맥 교환) 시의 디테일
- 인터럽트는 프로세스를 종료시키는 것이 아니라 **Preemption(선점, 강제로 뺏음)**하여 잠시 멈추는 것이다. CPU를 뺏길 때 현재 상태를 **PCB(Process Control Block)**에 저장한다.
- **저장 정보:** Program Counter(PC, 다음 실행할 메모리 주소), CPU Register 값, Stack Pointer 등.

**[다시 실행될 때 (재개 지점)]**
- 빼앗겼던 프로그램이 다시 CPU를 할당받으면, `while`문의 처음부터 다시 시작하는 것이 **절대 아니다.**
- PCB에 저장해둔 **PC(Program Counter)** 값을 읽어와서, 인터럽트가 걸렸던 정확한 기계어 명령어 그 다음 지점부터 이어서 실행을 재개한다.

---

### 🎤 [Backend Interview Point] Context Switching
**Q. Context Switching 오버헤드가 시스템에 미치는 영향은 무엇이며, 왜 발생하나요?**
- Context Switching이 발생하면 단순히 레지스터 값을 PCB에 저장하고 복원하는 연산 비용만 드는 것이 아닙니다. 
- 가장 큰 오버헤드는 **Cache Pollution(캐시 오염)**입니다. 기존 프로세스가 사용하며 올려둔 CPU Cache 메모리와 TLB(Translation Lookaside Buffer, 가상 메모리 주소 변환 캐시)가 새로운 프로세스로 넘어가면서 전부 무효화(Flush)되어야 합니다. 다시 캐시를 채우는(Cache Miss) 과정에서 막대한 성능 저하가 발생합니다. (이 때문에 프로세스보다 메모리를 공유하는 스레드/코루틴 단위의 스위칭이 훨씬 가볍습니다.)

## 2.4 I/O 수행 방식과 DMA (Direct Memory Access)

CPU가 I/O 컨트롤러에게 "디스크에서 데이터 읽어와"라고 명령(시스템 콜)을 내린 후, 데이터가 올 때까지 어떻게 행동할 것인가?

각 I/O 장치에는 장치를 전담하는 아주 작은 CPU 같은 존재인 **Device Controller(장치 제어기)**와 데이터를 임시로 저장하는 **Local Buffer**가 있다.

1. **동기식 입출력 (Synchronous I/O):**
   - I/O 요청 후 데이터가 도착할 때까지 해당 프로그램은 멈춰 대기(Blocked/Wait 상태)한다.
  - 이때 CPU까지 놀게 하면 극심한 낭비이므로, OS는 해당 프로세스에서 CPU를 뺏어 Ready 상태에 있는 다른 프로그램에게 넘겨준다.
2. **비동기식 입출력 (Asynchronous I/O):**
   - I/O 요청(시스템 콜)만 해두고, 데이터 읽기가 끝나든 말든 기다리지 않고 해당 프로그램이 곧바로 CPU를 유지한 채 다음 명령을 계속 수행한다.

### DMA (Direct Memory Access)의 등장 배경
**[DMA (Direct Memory Access)의 핵심 역할]**
- **문제:** I/O 장치(예: 키보드나 고속 네트워크 랜카드)가 1바이트를 받을 때마다 CPU에 인터럽트를 걸면, CPU는 하던 일을 멈추고 계속 인터럽트 처리를 해야 하므로 효율이 박살난다.
- **해결:** **메모리에 직접 접근할 수 있는 권한을 가진 컨트롤러인 DMA**를 둔다.
- **방식:** CPU는 DMA에게 "디스크에서 데이터 1블록 읽어와"라고 지시만 하고 자기 할 일을 한다. DMA가 Device Controller의 Local Buffer에서 데이터를 빼내어 **메모리에 블록(Block) 단위로 직접 복사**한다. 복사가 완전히 끝나면, 그때서야 DMA가 CPU에게 **딱 한 번만 인터럽트**를 건다. (CPU의 인터럽트 당하는 빈도를 획기적으로 줄여줌)

## 2.5 프로그램의 실행과 메모리 구조 Deep Dive

### 저장장치 계층 구조 (Storage Hierarchy)
- **휘발성 (Primary):** `Register` -> `Cache` -> `Main Memory(RAM)`. 여기까지는 CPU가 바이트 단위로 직접 주소를 지정해 연산할 수 있다. (Byte-addressable)
- **비휘발성 (Secondary):** `Magnetic Disk(HDD/SSD)` -> `Optical Disk` -> `Magnetic Tape`. CPU가 직접 연산할 수 없으며, I/O 컨트롤러를 통해 블록 단위로 데이터를 메모리에 올려야만 CPU가 읽을 수 있다.

### 가상 메모리 (Virtual Memory)와 주소 변환
- 실행 파일을 실행하면 물리적 메모리에 곧바로 올라가지 않는다.
- 프로세스마다 `0번지`부터 시작하는 자신만의 독자적인 주소 공간인 **가상 메모리(Address Space)**가 생성된다. (Code, Data, Stack으로 구성)
- **주소 변환(Address Translation):** 당장 CPU가 실행해야 하는 핵심 코드만 하드웨어 장치(MMU)의 지원을 받아 물리적 메모리 주소로 변환되어 올라간다.
- **Swap Area:** 당장 쓰이지 않는 부분은 디스크의 **Swap Area**에 내려둔다. (Swap Area는 비휘발성 디스크 물리 매체를 사용하지만, 전원이 나가면 의미가 없는 데이터를 담고 있으므로 '메모리의 연장 공간'으로 취급해야 한다. 영구 저장용 File System과는 다르다.)

### 커널(OS) 주소 공간의 내부 구조
운영체제도 결국 하나의 소프트웨어이므로 가상 메모리에 0번지부터 공간을 갖는다.
- **Code (커널 코드):** 인터럽트 처리 코드, 시스템 콜 처리 코드, 자원 관리(CPU 스케줄링, 메모리 관리) 로직.
- **Data:** CPU, 메모리 등 모든 하드웨어를 관리하는 자료구조. 그리고 현재 실행 중인 각 프로세스의 상태를 관리하기 위한 **PCB(Process Control Block)** 자료구조가 여기에 배열이나 링크드 리스트 형태로 저장된다.
- **Stack:** 커널 코드도 함수 호출을 하므로 스택이 필요하다. 단, 보안과 추적을 위해 **어떤 프로세스가 시스템 콜을 호출했는지에 따라 각 프로세스별로 별도의 커널 스택을 나누어 유지**한다. (Process A의 커널 스택, Process B의 커널 스택)

### 함수 호출과 Mode 변경 흐름
- **사용자/라이브러리 함수 호출:** 내 프로세스의 주소 공간 내(Code 영역)에서 점프함. (Mode bit 1 유지)
- **시스템 콜 호출 (커널 함수):** 내 프로세스 공간을 벗어나 OS의 주소 공간(커널 Code)으로 넘어가야 함. 반드시 소프트웨어 인터럽트(Trap)를 발생시켜 Mode bit를 0으로 바꿔야만 커널 공간에 접근할 수 있다. 작업이 끝나면 다시 리턴하며 Mode bit 1로 복귀.

### 🎤 [Backend Interview Point] 메모리 영역 심화
**Q. 운영체제의 Data 영역에 프로세스 제어 블록(PCB)이 저장된다고 하셨습니다. PCB에는 구체적으로 어떤 정보들이 들어가며, 왜 중요할까요?**
- PCB는 OS가 프로세스를 관리하기 위한 호적등본(Metadata) 같은 역할을 합니다. 
- **저장 내용:** 현재 프로세스의 상태(Ready, Running, Blocked), Program Counter 값, CPU 레지스터 값들, 프로세스 ID(PID), 메모리 제한 정보, 열어둔 파일(File Descriptors) 목록 등이 들어갑니다.
- **중요성:** Timer Interrupt 등으로 인해 Context Switching이 발생할 때, CPU를 뺏기는 프로세스의 현재 작업 상태를 잃어버리지 않고 안전하게 보관하는 유일한 저장소가 바로 커널 Data 영역의 PCB이기 때문입니다. PCB가 없다면 멀티태스킹 자체가 불가능합니다.
