# 운영체제 면접 스터디 Q&A
---

## Q1.
Device Controller는 CPU처럼 범용 연산을 수행할 수 있나요?

---

## A1.

불가능합니다.

Device Controller는 CPU처럼 보일 수 있지만,
실제 CPU처럼 범용 연산을 수행하는 장치는 아닙니다.

정확히는 특정 I/O 장치를 제어하기 위한 전용 제어기입니다.

내부에 간단한 프로세서(마이크로컨트롤러)나 펌웨어가 들어있는 경우가 많지만,
목적이 제한되어 있습니다.

즉:
- CPU → 범용 연산 수행
- Device Controller → 특정 장치 제어 전용

이라는 차이가 있습니다.

---

### 추가설명 예시

- SSD Controller
  - 플래시 메모리 관리
  - wear leveling
  - garbage collection 수행

- GPU 내부 Controller
  - 그래픽 작업 스케줄링

- NIC(Network Interface Card)
  - 네트워크 패킷 처리 일부 수행

---

## Q1 추가 꼬리질문

### Q.
그렇다면 왜 Device Controller가 필요한가요?

### A.
CPU가 모든 I/O 장치를 직접 제어하면 부담이 매우 커지기 때문입니다.

따라서 Device Controller가:
- 장치와의 세부 통신 처리
- 데이터 버퍼 관리
- interrupt 발생

등을 대신 수행하여 CPU 부담을 줄여줍니다.

## Q2: 디스크에 있는 대용량 텍스트 파일을 읽어와서 처리하는 코드를 작성한다고 가정해 보겠습니다. 1바이트씩 1만 번 읽는 방식과, 1만 바이트 크기의 버퍼(Buffer)를 두고 한 번에 읽는 방식 중 어떤 것이 성능에 유리할까요? 운영체제의 시스템 콜(System Call) 관점에서 원리를 설명해 주세요.

### A2: 네, 버퍼를 활용하여 한 번에 읽는 방식이 성능에 압도적으로 유리합니다.

- 일반적인 사용자 프로그램은 User Mode(1) 상태이므로 디스크에 직접 접근할 권한이 없습니다. 따라서 디스크를 읽으려면 OS에게 시스템 콜(System Call)을 호출하여 Kernel Mode(0)로 권한을 전환(Mode Switch)해야 합니다. 하지만 이 전환 과정 자체가 시스템에 막대한 오버헤드를 발생시킵니다.

- 만약 1바이트씩 1만 번을 읽는다면 이 무거운 시스템 콜이 1만 번이나 발생하여 CPU 성능이 크게 저하됩니다. 반면 1만 바이트 크기의 버퍼를 사용하게 되면, 시스템 콜 호출 횟수를 단 1번으로 획기적으로 줄일 수 있기 때문에 결과적으로 극적인 성능 최적화를 이루어낼 수 있습니다."

## Q3:"운영체제에서 Context Switching(문맥 교환)이 발생할 때 단순히 레지스터 값을 저장하고 불러오는 것 외에 시스템에 미치는 가장 큰 성능적 오버헤드는 무엇인가요? 그리고 이 오버헤드를 줄이기 위해 최근 백엔드 환경에서는 어떤 접근을 하고 있나요?"


### A3:"문맥 교환 시 단순히 레지스터 값을 읽고 쓰는 비용을 넘어서, 가장 치명적인 오버헤드는 바로 캐시 오염(Cache Pollution)입니다. 

- 문맥 교환이 잦아지면 기존 프로세스가 사용하던 CPU 캐시 메모리와 가상 메모리 주소 변환 캐시인 TLB가 계속 Flush(초기화)되어야 하므로 막대한 성능 저하가 발생합니다.

- 기존의 전통적인 동기/블로킹 웹 서버는 I/O 작업이 발생할 때마다 해당 스레드가 대기 상태에 빠지고, 이때마다 OS가 강제로 문맥 교환을 발생시켜 이 오버헤드를 정통으로 맞았습니다.

- 이를 해결하기 위한 실무적 접근으로 최근에는 Node.js나 Spring WebFlux 같은 비동기/논블로킹(Async/Non-blocking) 기반의 프레임워크를 적극 활용하고 있습니다. 이 환경에서는 스레드가 I/O 작업을 OS에 위임한 뒤 대기하지 않고 즉시 다음 요청을 처리합니다. 즉, 스레드가 멈추지 않으므로 OS의 불필요한 문맥 교환이 획기적으로 줄어들고, CPU 캐시가 효율적으로 유지되어 고동시성 트래픽을 아주 가볍게 처리할 수 있습니다."

## Q4: 만약 배포한 서버 애플리케이션 코드에 논리적 버그가 있어서 while(true) 무한 루프에 빠졌다고 가정해 봅시다. 이 프로그램 하나 때문에 해당 서버의 CPU가 영원히 독점되어 다른 프로그램들까지 멈추는 일은 발생하지 않는데, 그 이유를 하드웨어 인터럽트와 PCB를 연관 지어 설명해 주시겠어요?


### A4: 네, 프로그램이 무한 루프에 빠지더라도 하드웨어 장치인 Timer가 존재하기 때문에 시스템이 멈추지 않습니다.

- Timer는 설정된 시간(Time Quantum)마다 하드웨어 인터럽트를 발생시키는데, 이때 Mode bit가 0으로 바뀌면서 CPU 제어권이 해당 프로그램에서 운영체제(OS)로 강제 회수(Preemption)됩니다.

- 운영체제는 CPU를 다른 프로그램에게 넘겨주기 전에, 무한 루프를 돌고 있던 프로그램이 다음에 실행할 기계어 명령어의 주소(Program Counter)와 레지스터 상태를 커널의 Data 영역에 있는 해당 프로세스의 PCB(Process Control Block)에 저장해 둡니다. 덕분에 특정 프로그램이 CPU를 영구적으로 독점하는 것을 막고 안전하게 멀티태스킹을 유지할 수 있습니다."
