1. 스레드(Thread)의 장점을 말해보시오.

스레드는 하나의 프로세스 안에서 여러 실행 흐름을 만드는 단위입니다. 스레드의 주요 장점은 네 가지로 정리할 수 있습니다.

첫째, 응답성(Responsiveness)입니다. 하나의 스레드가 I/O 작업 등으로 블록되더라도 같은 프로세스의 다른 스레드가 계속 실행될 수 있어 사용자 입장에서 응답이 끊기지 않습니다.

둘째, 자원 공유(Resource Sharing)입니다. 같은 프로세스 내 스레드들은 코드, 데이터, 힙 영역을 공유하므로 프로세스를 여러 개 만드는 것보다 메모리를 훨씬 절약할 수 있습니다.

셋째, 경제성(Economy)입니다. 프로세스 생성과 Context Switch는 비용이 크지만, 스레드 생성과 스레드 간 전환은 훨씬 가볍습니다.

넷째, 멀티프로세서 활용(Utilization of MP Architectures)입니다. 멀티코어 환경에서 스레드들을 각 CPU에 병렬로 실행시켜 성능을 극대화할 수 있습니다.

2. 커널 관점에서 Race Condition 문제를 말해보시오.
커널 내부에서 Race Condition이 발생할 수 있는 상황은 크게 세 가지입니다.

첫째, 커널 코드 실행 중 인터럽트가 발생하는 경우입니다. 커널이 공유 데이터를 수정하는 도중 인터럽트 핸들러가 같은 데이터를 수정하면 결과가 뒤바뀔 수 있습니다. 해결책으로는 커널이 공유 데이터를 수정하는 동안 인터럽트를 disable합니다.

둘째, 프로세스가 커널 모드로 실행 중 Context Switch가 발생하는 경우입니다. 프로세스 A가 시스템 콜로 커널 모드에서 작업하던 중 CPU를 빼앗기고, 프로세스 B가 같은 데이터를 수정하면 A가 복귀했을 때 잘못된 값을 덮어씁니다. 해결책은 커널 모드에서는 CPU를 preempt하지 않는 것입니다.

셋째, 멀티프로세서 환경입니다. 여러 CPU가 동시에 커널 내 같은 데이터에 접근할 수 있어 인터럽트 disable만으로는 해결되지 않습니다. 이 경우 해당 데이터에 lock을 거는 방식으로 해결합니다.


3. 단일 CPU 구조에서 Timer Interrupt가 필요한 이유에 대해 간단히 설명해주세요.
단일 CPU 환경에서 하나의 프로그램이 CPU를 독점하면 다른 프로그램은 영원히 실행될 수 없습니다. 이를 방지하기 위해 Timer Interrupt가 필요합니다.

운영체제는 프로그램에 CPU를 넘겨줄 때 Timer에 일정 시간을 설정합니다. 해당 시간이 만료되면 Timer가 자동으로 Interrupt를 발생시켜 CPU 제어권이 다시 운영체제로 돌아옵니다. 운영체제는 이를 이용해 다음 프로세스에게 CPU를 넘겨주는 스케줄링을 수행합니다.

즉, Timer Interrupt는 특정 프로그램의 CPU 독점을 방지하고, 운영체제가 CPU 제어권을 주기적으로 회수하기 위한 핵심 메커니즘입니다.

4. 라운드 로빈 알고리즘에서 Time Quantum의 변화가 미치는 영향은?

Time Quantum은 각 프로세스가 한 번에 CPU를 사용할 수 있는 최대 시간 단위입니다.

Time Quantum이 매우 클 경우, 각 프로세스가 끝날 때까지 다 사용하게 되어 사실상 FCFS(선착순) 알고리즘과 동일하게 동작합니다. 응답 시간이 길어지는 단점이 있습니다.

Time Quantum이 매우 작을 경우, 프로세스 전환이 너무 자주 발생하여 Context Switch 오버헤드가 폭증합니다. 실제 작업보다 스위칭에 더 많은 시간을 쓰게 되어 전체 성능이 저하됩니다.

따라서 적절한 Time Quantum을 설정하는 것이 중요하며, 일반적으로 CPU Burst의 80% 정도가 Time Quantum보다 짧도록 설정하는 것이 권장됩니다. 보통 10ms~100ms 사이 값을 사용합니다.

5. Critical Section 문제를 해결하는 3가지 조건은?

Critical Section 문제를 올바르게 해결하려면 다음 세 가지 조건을 반드시 만족해야 합니다.

첫째, Mutual Exclusion(상호 배제)입니다. 한 프로세스가 Critical Section을 실행 중이라면 다른 프로세스는 절대 동시에 진입할 수 없어야 합니다.

둘째, Progress(진행)입니다. Critical Section을 실행 중인 프로세스가 없고, 진입을 원하는 프로세스가 있다면 그 중 하나가 반드시 진입할 수 있어야 합니다. 즉, 아무도 없는데 진입이 막히는 상황이 없어야 합니다.

셋째, Bounded Waiting(한정 대기)입니다. 하나의 프로세스가 Critical Section 진입을 요청한 이후, 다른 프로세스들이 진입하는 횟수에 한계가 있어야 합니다. 즉, 무한정 기다리는 상황(Starvation)이 발생해서는 안 됩니다.


---

조건 빠지면발생하는 문제
Mutual Exclusion 없으면동시 접근 → 데이터 오염
Progress 없으면아무도 없는데 전부 대기 → 
DeadlockBounded Waiting 없으면특정 프로세스만 계속 대기 → Starvation 발생

6. Context Switching 시 무엇이 PCB에 저장되는가?
- PCB(Process Control Block)는 운영체제가 각 프로세스를 관리하기 위해 유지하는 자료구조야.

Context Switch는 CPU를 한 프로세스에서 다른 프로세스로 넘겨줄 때 발생하며, 이때 운영체제는 현재 프로세스의 상태를 PCB(Process Control Block)에 저장합니다.

PCB에 저장되는 정보는 크게 세 가지입니다.

첫째, CPU 레지스터 값입니다. Program Counter(다음 실행할 명령어 주소), 스택 포인터, 범용 레지스터 등 CPU의 현재 상태를 모두 저장합니다.

둘째, 프로세스 상태(Process State)입니다. Running, Ready, Blocked 등 현재 프로세스가 어떤 상태인지를 기록합니다.

셋째, 메모리 관련 정보입니다. 프로세스가 사용 중인 메모리 영역 정보(페이지 테이블 등)를 저장합니다.

이후 해당 프로세스가 다시 CPU를 받으면, PCB에 저장된 값을 복원하여 중단된 지점부터 정확히 이어서 실행합니다.




- 저장되는 것들, 왜 각각 필요한가?
- CPU 레지스터 & Program Counter

→ "어디까지 실행했는지" 기록. 없으면 다음에 CPU 돌려받았을 때 처음부터 다시 실행해야 함.

- 프로세스 상태 (Running / Ready / Blocked)

→ 스케줄러가 "이 프로세스 지금 실행 가능한가?"를 판단하는 기준.

- 메모리 정보 (페이지 테이블 등)

→ 프로세스가 어느 메모리 공간을 쓰고 있는지. 없으면 다른 프로세스 메모리를 침범할 수 있음.
