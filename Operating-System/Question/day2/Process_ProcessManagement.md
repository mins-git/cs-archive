# 문제 1
1. 실시간으로 수많은 사용자의 요청(네트워크 I/O)을 지연 없이 처리해야 하는 백엔드 서버를 설계한다고 가정해 보겠습니다. 이러한 환경에서 동기식(Synchronous) I/O와 비동기식(Asynchronous) I/O 중 어떤 방식을 선택하는 것이 시스템 성능상 유리하며, 운영체제의 스레드 활용 및 문맥 교환(Context Switching) 관점에서 그 이유를 설명해 주세요.

- "해당 환경에서는 비동기식(Asynchronous) I/O를 선택하는 것이 압도적으로 유리합니다.
동기식 I/O를 사용하면 커널에 디스크나 네트워크 작업을 요청한 후, 완료될 때까지 해당 스레드가 Blocked(대기) 상태에 빠져 아무것도 할 수 없습니다. 트래픽이 몰리면 수많은 스레드를 새로 생성해야 하므로 물리적인 메모리 낭비와 극심한 문맥 교환 오버헤드가 발생합니다.
반면, 비동기식을 사용하면 I/O 작업을 커널에 위임한 후 스레드가 곧바로 다른 클라이언트의 요청을 이어서 처리할 수 있습니다. 즉, 적은 수의 스레드만으로도 실시간 주식 호가 데이터 스트리밍이나 대규모 매수/매도 트래픽을 병목 없이 처리할 수 있어 CPU 효율이 극대화됩니다."


-> 꼬리 질문 1-1: 비동기식을 선택하면 프로세스(또는 스레드)가 대기 큐(Blocked Queue)에서 멈춰 있지 않고 하던 일을 계속한다고 하셨습니다. 그렇다면 커널이 하드웨어 I/O 작업을 끝냈을 때, 이를 어떻게 인지하고 이후 프로세스 상태는 어느 큐로 이동하게 되나요? [비동기 I/O 완료 시점 인지와 큐 이동]

- "비동기 I/O 작업이 완료되면, 네트워크 카드나 디스크 같은 하드웨어가 CPU에 '인터럽트(Interrupt)' 신호를 보냅니다. 커널은 이 인터럽트를 감지하여 작업 완료를 인지하고, 해당 결과를 기다리며 대기 큐(Blocked Queue)에 머물거나 지연 상태(Suspended)에 있던 프로세스를 다시 레디 큐(Ready Queue)로 이동시킵니다. 이후 스케줄러에 의해 다시 CPU를 할당받으면 멈췄던 로직을 이어서 실행하게 됩니다."

-> 꼬리 질문 1-2: 성능을 높이겠다고 서버에 너무 많은 스레드를 띄워두면, 커널의 스케줄러와 디스패처에 의해 문맥 교환(Context Switch)이 무수히 발생하게 됩니다. 이때 PCB(Process Control Block)를 다루는 과정에서 시스템에 구체적으로 어떤 물리적 부담(오버헤드)이 발생할 수 있나요? [문맥 교환의 물리적 오버헤드]

- "가장 직관적인 오버헤드는 기존 스레드의 PCB(또는 TCB)를 메모리에 저장하고 새 데이터를 로드하는 연산 비용입니다.
하지만 더 치명적인 것은 **CPU 캐시 무효화(Cache Invalidation)**에서 오는 성능 저하입니다. 문맥 교환이 일어나면 CPU 내부에 기껏 세팅해 둔 이전 프로세스의 캐시 데이터와 메모리 매핑 정보(TLB)가 쓸모없어집니다. 새로운 프로세스가 CPU를 잡으면 필요한 데이터를 메모리에서 처음부터 다시 끌어와야 하는 '캐시 미스(Cache Miss)'가 대량으로 발생하게 되며, 이것이 서버 응답 지연의 핵심 원인이 됩니다."

# 문제 2
2. 리눅스 기반의 서버 환경에서 멀티 프로세스 아키텍처를 운영하다 보면, 자원 관리에 실패하여 어느 순간 시스템에 **'좀비 프로세스(Zombie Process)'**가 쌓이고 결국 서버가 완전히 마비되는 장애가 발생할 수 있습니다. 좀비 프로세스가 정확히 무엇이며, 부모-자식 프로세스 모델과 시스템 콜(exit, wait)의 동작 원리 관점에서 왜 이런 현상이 발생하는지 설명해 주세요.

- "좀비 프로세스는 자식 프로세스가 할 일을 모두 마치고 exit() 시스템 콜을 호출해 스스로 종료했음에도 불구하고, 부모 프로세스가 wait() 시스템 콜을 호출하지 않아 발생합니다.
운영체제는 자식이 종료될 때 부모에게 종료 상태(Exit status)를 전달하기 위해, 자식의 실제 메모리 공간은 비우더라도 자식의 신분증인 PID와 최소한의 정보가 담긴 PCB를 메모리에 남겨둡니다. 부모가 이를 끝끝내 확인(수거)하지 않아 이 껍데기 정보가 시스템에 계속 누적되는 상태를 좀비 프로세스라고 합니다."


-> 꼬리 질문 2-1: 좀비 프로세스가 계속 쌓일 때 서버가 새로운 프로세스를 띄우지 못하고 마비되는 근본적인 원인은 무엇인가요? (단순히 RAM 메모리 용량이 꽉 차서인지, 아니면 OS가 관리하는 다른 자원의 고갈 때문인지 정확히 설명해 주세요.) [서버 마비의 근본적 원인]

- "단순히 메모리(RAM) 용량이 가득 차서가 아니라, 커널이 발급할 수 있는 프로세스 ID(PID)가 고갈되기 때문입니다.
리눅스 환경에서 부여할 수 있는 최대 PID 개수는 한정되어 있습니다(보통 약 3만 2천 개). 좀비 프로세스들이 이 PID를 반납하지 않고 모두 쥐고 있으면, 새로운 클라이언트의 요청이 들어와도 커널이 fork()를 통해 새로운 자식 프로세스나 스레드를 생성할 수 없게 되어 결국 서버 전체가 요청을 거부하는 마비 상태에 빠집니다."

-> 꼬리 질문 2-2: 반대로 부모 프로세스가 예기치 못한 에러(Abort)로 자식보다 먼저 종료되어 버리는 '고아 프로세스(Orphan Process)' 상황이 발생한다면, 운영체제는 주방에 덩그러니 남겨진 이 자식 프로세스들을 어떻게 안전하게 수거(처리)하나요? [고아 프로세스 발생 시 운영체제의 대처]

- "부모 프로세스가 치명적인 오류(Abort) 등으로 자식보다 먼저 비정상 종료되면, 시스템에는 부모를 잃은 고아 프로세스들이 남게 됩니다.
이때 운영체제는 시스템의 최상단 조상인 **PID 1번 프로세스(리눅스의 init 또는 systemd)**를 내세워 이 고아 프로세스들의 새로운 부모로 지정하는 '입양(Re-parenting)' 과정을 거칩니다. 이후 고아 프로세스가 작업을 마치고 exit()을 호출하면, 1번 프로세스가 대신 wait()를 처리해주어 좀비가 되는 것을 막고 PCB와 PID를 안전하게 회수합니다."
