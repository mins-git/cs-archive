## 1. Process 1: 프로세스의 개념과 상태

### 1) 프로세스의 개념 (Concept of Process)
* **정의:** 실행 중인 프로그램(Program in execution)을 의미합니다.
* **특징:** 디스크에 정적으로 존재하던 프로그램이 메모리에 올라와 CPU를 할당받고 실행되는 동적인 상태를 말합니다.
* **프로세스의 문맥 (Process Context):** 프로세스가 현재 어느 단계까지 실행되었는지 정확히 규명하기 위해 필요한 모든 정보입니다. CPU의 시분할 공유로 인해 프로세스가 교체될 때 기존 상태를 보존하기 위해 필수적입니다.
  * **하드웨어 문맥:** Program Counter(PC), 범용 레거스터(Register) 값 등 CPU의 현재 상태.
  * **프로세스의 주소 공간:** 가상 메모리의 Code, Data, Stack, Heap 영역의 상태.
  * **커널 측 문맥:** OS가 프로세스를 관리하기 위해 두는 영역으로, PCB(Process Control Block) 및 Kernel Stack 상태.

### 2) 프로세스의 상태 (Process State)
프로세스의 상태는 CPU 할당 및 외부 이벤트에 따라 끊임없이 변화합니다.

* **실행 (Running):** CPU를 잡고 명령어(Instruction)를 수행 중인 상태.
* **준비 (Ready):** CPU만 할당받으면 즉시 실행할 수 있는 상태 (메모리 등 다른 조건은 모두 충족된 상태).
* **봉쇄 / 대기 / 수면 (Blocked / Wait / Sleep):** CPU를 주어도 당장 명령을 수행할 수 없는 상태. I/O 작업 완료 등 외부 이벤트가 발생해야 해제됩니다.
  - ex) 디스크에서 file을 읽어와야 하는 경우.
* **시작 (New) & 종료 (Terminated):** 프로세스가 생성 중이거나, 실행이 끝나고 정리하는 임시 상태.
* **중단 상태 (Suspended / Stopped):** 외부적인 이유(메모리 부족, 사용자의 일시정지 제어 등)로 프로세스의 실행이 통째로 정지되어 **메모리에서 통째로 디스크(Swap Out)**로 쫓겨난 상태.
  * *Suspended Ready:* 준비 상태에서 Swap Out 됨.
  * *Suspended Blocked:* 봉쇄 상태에서 Swap Out 됨.

### 3) 프로세스 상태도 (Process State Diagram)
```
 [New] ---> [Ready] <======(Dispatch / Interrupt)======> [Running] ---> [Terminated]
              ^                                             |
              |                                             v
              +--- (I/O 완료 / Event 발생) <--- [Blocked / Wait]
```

### 4) 프로세스 제어 블록 (PCB, Process Control Block)
- pcb는 커널공간에 보관된다. 
  - 일반 프로그램(사용자 프로세스)이 남의 정보(PCB)를 훔쳐보거나 마음대로 조작하면 컴퓨터가 망가지기 때문에, 오직 매니저인 커널만이 자신의 공간에 꽁꽁 숨겨두고 관리하는 것이죠.
* **정의:** 운영체제가 각 프로세스를 관리하기 위해 프로세스당 하나씩 유지하는 커널 내의 정보 블록입니다.
* **구조 및 포함 정보:**
  1. **OS가 관리상 사용하는 정보:** Process ID(PID), Process State, Scheduling Priority.
  2. **CPU 수행 관련 하드웨어 상태:** Program Counter(PC), Registers.
  3. **메모리 관리 정보:** Code, Data, Stack의 위치 및 Page Table, Segment Table 정보.
  4. **자원 사용 정보:** 열린 파일 디스크립터(Open file descriptors), 사용중인 입출력 장치 목록 등.

### 5) 문맥 교환 (Context Switch) -> 디스패처가 처리함
* **정의:** CPU를 한 프로세스에서 다른 프로세스로 넘겨주는 과정을 말합니다.
* **수행 과정:**
  1. CPU를 반납하는 프로세스의 하드웨어 문맥(PC, 레지스터 등)을 해당 프로세스의 PCB에 저장합니다.
  2. 새로 CPU를 할당받을 프로세스의 PCB로부터 문맥을 복원하여 CPU 레지스터에 탑재합니다.
* **주의점:** * 유저 모드에서 시스템 콜(System Call)이나 인터럽트(Interrupt)가 발생하여 커널 모드로 진입했다가 다시 동일 프로세스로 돌아오는 것은 문맥 교환이 아닙니다. (단순 Mode Switch)
  * 다른 프로세스로 CPU 제어권이 넘어갈 때만 **문맥 교환(Context Switch)**이라고 하며, 이때 Cache Memory Flush 등 상당한 오버헤드(Overhead)가 발생합니다.
    - 디스패처가 0.01초마다 기존 요리사를 쫓아내고 다음 요리사를 가스레인지에 앉히는 작업을 미친 듯이 빠른 속도로 반복합니다. (이 방식을 라운드 로빈(Round Robin) 스케줄링이라고 합니다.)
      - 결과적으로 CPU(가스레인지)는 하나지만, 교체 속도가 인간의 눈에는 보이지 않을 정도로 너무 빠르기 때문에 우리는 "음악을 들으면서, 웹서핑을 하고, 카카오톡을 동시에 하고 있네!"라고 착각하게 되는 것입니다!

### 6) 프로세스 스케줄링을 위한 큐 (Scheduling Queues)
OS는 프로세스들을 관리하기 위해 연결 리스트(Linked List) 형태의 큐를 사용합니다.
* **Job Queue:** 시스템 안의 모든 프로세스 집합 (메모리 유무 무관).
* **Ready Queue:** 현재 메모리에 올라와서 CPU를 잡기 위해 대기하는 프로세스 집합.
* **Device Queue:** 특정 I/O 장치(Disk, Keyboard 등)의 서비스를 받기 위해 대기하는 프로세스 집합.

### 7) 스케줄러 (Scheduler)
* **단기 스케줄러 (Short-term Scheduler / CPU Scheduler):** Ready Queue에 있는 프로세스 중 어떤 것에 CPU를 줄지 결정합니다. 밀리초(ms) 단위로 매우 자주 호출되므로 빨라야 합니다.
* **장기 스케줄러 (Long-term Scheduler / Job Scheduler):** 어떤 프로세스를 메모리에 올릴지(Ready Queue로 보낼지) 결정합니다. **다중프로그래밍 학위(Degree of Multiprogramming)**를 제어합니다. 현대 시분할 시스템(Time-sharing system)에서는 보통 장기 스케줄러를 두지 않고 무조건 메모리에 올린 후 중기 스케줄러로 조절합니다.
* **중기 스케줄러 (Medium-term Scheduler / Swapper):** 메모리에 너무 많은 프로세스가 올라와 있을 때, 일부 프로세스를 통째로 메모리에서 디스크(Swap-out)로 쫓아내어 다중프로그래밍 학위를 동적으로 조절합니다.

### 8) '지연(Suspended)'의 진짜 의미: 메모리(RAM)에서 쫓겨나다
- 지금까지 배운 Ready, Running, Blocked 상태의 프로세스들은 모두 메모리(RAM)에 자리를 잡고 있었습니다.

- 하지만 지연(Suspended) 상태가 되었다는 것은, 메모리 공간이 부족해서 프로세스가 하드디스크(정확히는 Swap Area라는 디스크 공간)로 쫓겨났다는 것을 의미합니다. 이를 전문 용어로 스왑 아웃(Swap-out)이라고 합니다.

 - CPU는 하드디스크에 있는 코드를 직접 읽을 수 없기 때문에, 지연 상태의 프로세스는 CPU를 얻을 자격이 박탈된 상태입니다.

 - 1. Suspended 상태는 언제, 왜 생길까?
    - 보통 크게 두 가지 이유로 발생합니다.

    - ① 메모리(RAM)가 꽉 찼을 때 (OS의 결정):
      컴퓨터에 프로그램(게임, 크롬, 개발 툴 등)을 너무 많이 켜서 RAM이 부족해지면, 운영체제(커널)는 결단을 내립니다. "지금 당장 일을 안 하고 멈춰 있는 프로세스(Blocked)들을 하드디스크로 쫓아내고, 그 자리에 당장 실행해야 하는 프로세스를 올리자!" 하고 프로세스를 디스크로 던져버립니다.

    - ② 외부에서 정지 명령을 내렸을 때 (사용자/디버거의 결정):
      리눅스 터미널에서 프로그램을 실행하다가 Ctrl + Z를 누르면 프로세스가 잠시 멈춥니다. 또는 개발자가 코드를 한 줄씩 확인하기 위해 '디버거(Debugger)'로 프로세스를 중단시켰을 때도 지연 상태가 됩니다.

  - 2. Suspended 상태의 두 가지 종류와 이동 경로
    쫓겨날 당시의 상태에 따라 두 가지로 나뉩니다.

    - 지연 대기 (Suspended Blocked): I/O(입출력) 작업을 기다리던 중(Blocked)에 메모리에서 쫓겨나 디스크로 간 상태입니다.

    - 지연 준비 (Suspended Ready): 당장 CPU만 주면 일할 수 있는 상태(Ready)였는데, 메모리가 부족해서 억울하게 디스크로 쫓겨난 상태입니다.

  - 어떻게 다시 메모리(Ready, Blocked)로 올라올까?

    - 디스크에 있던 친구들이 다시 메모리로 돌아오는 과정을 스왑 인(Swap-in)이라고 합니다.

    - 1. 지연 대기 큐에서 I/O가 끝난 경우:
      디스크에 쫓겨나 있는 동안에도 키보드 입력이나 파일 읽기가 끝날 수 있습니다. 그러면 하드웨어가 인터럽트를 보내고, 커널은 이 프로세스를 지연 대기(Suspended Blocked)에서 지연 준비(Suspended Ready) 상태로 바꿉니다. (아직 디스크에 있지만, 메모리로 돌아갈 준비가 된 것입니다.)

    - 2. 메모리에 여유가 생기거나 외부에서 깨웠을 때:
      다른 프로그램이 종료되어 RAM에 빈자리가 생기거나, 사용자가 "다시 시작해!"라는 명령(Ctrl + Z로 멈춘 경우 fg나 bg 명령어 사용)을 내리면, 커널은 지연 준비(Suspended Ready) 상태의 프로세스를 다시 메모리의 레디 큐(Ready Queue)로 엄연히 복귀시킵니다.

## 2. Process 2 & 3: 입출력 메커니즘과 스레드 (Thread)

### 1) 동기식 입출력과 비동기식 입출력 (Synchronous vs Asynchronous I/O)
* **동기식 입출력 (Synchronous I/O):**
  * I/O 요청을 보낸 후, 입출력 작업이 완료되어 데이터가 커널/유저 영역에 도달할 때까지 **프로세스가 기다리는 방식**입니다.
  * 해당 프로세스는 I/O 작업 동안 Blocked 상태가 되며 CPU를 반납합니다. OS는 Ready Queue의 다른 프로세스에게 CPU를 넘깁니다.
* **비동기식 입출력 (Asynchronous I/O):**
  * I/O 요청을 보낸 후, 결과가 나오기를 기다리지 않고 **즉시 CPU 제어권을 다시 얻어 다음 명령어들을 계속 수행하는 방식**입니다.
  * I/O 작업이 끝나면 인터럽트를 통해 완료 통보를 받습니다.

### 2) 스레드 (Thread)
* **정의:** 프로세스 내부의 CPU 수행 단위 또는 실행 흐름을 의미합니다. (Lightweight Process라고도 불림)
* **프로세스 내 공유 공간 (Shared Areas):**
  * 하나의 프로세스 안에서 **Code, Data, OS Resources(오픈 파일 등)** 영역은 모든 스레드가 공유합니다.
* **스레드 개별 공간 (Independent Areas):**
  * 각 스레드마다 독립적인 실행 흐름을 가져야 하므로 **Program Counter(PC), Register Set, Stack** 영역은 스레드별로 별도 관리됩니다.

### 3) 스레드의 장점 (Benefits of Threads)
1. **응답성 (Responsiveness):** 웹 브라우저에서 하나의 스레드가 이미지를 다운로드(I/O Block)받는 동안, 다른 스레드가 텍스트나 사용자 상호작용을 처리하여 응답 속도를 높일 수 있습니다.
2. **자원 공유 (Resource Sharing):** 프로세스의 자원과 메모리를 공유하므로 하나의 메모리 공간 내에서 효율적으로 작업할 수 있습니다.
3. **경제성 (Economy):** 프로세스를 새로 생성하거나 프로세스 간 문맥 교환을 하는 것보다, 스레드를 생성하고 스레드 간 전환(Context Switch)하는 것이 오버헤드가 훨씬 적습니다.
4. **멀티프로세서 구조 활용:** 각각의 스레드가 서로 다른 CPU(코어)에서 병렬로 실행되어 처리 속도를 극대화할 수 있습니다.

### 4) 스레드의 구현 방식 (Implementation of Threads)
* **커널 수준 스레드 (Kernel-level Threads):** 운영체제 커널이 스레드의 존재를 알고 있으며, 직접 스케줄링 및 관리를 수행합니다. 하나의 스레드가 Block 되어도 다른 스레드는 계속 실행 가능합니다.
* **사용자 수준 스레드 (User-level Threads):** 커널은 스레드의 존재를 모른 채, 사용자 영역의 스레드 라이브러리를 통해 스레드를 관리합니다. 커널 눈에는 단일 프로세스로 보이기 때문에, 하나의 스레드가 Block 되면 프로세스 전체가 Block 되는 단점이 있으나 전환 속도가 빠릅니다.
